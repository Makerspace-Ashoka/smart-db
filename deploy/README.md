# Deployment

This directory contains the production stack for the lab host:

- `compose.yaml`: Smart DB gateway, middleware, and local Part-DB on an isolated bridge network with explicit published ports
- `Dockerfile.gateway`: builds the frontend and serves it through Caddy with HTTPS
- `Dockerfile.middleware`: runs the Fastify middleware on Node 24
- `config/*.env.example`: non-secret environment templates
- `scripts/backup.sh`: SQLite-aware backups plus media archives. Secret-bearing config and TLS keys are excluded by default.
- `systemd/`: timer unit for nightly backups

## Runtime layout

- `deploy/state/smartdb/data`: Smart DB SQLite database
- `deploy/state/partdb/db`: Part-DB SQLite database
- `deploy/state/partdb/uploads`: Part-DB uploads and exports
- `deploy/state/partdb/public-media`: Part-DB public media
- `deploy/state/caddy`: Caddy certificates and config
- `deploy/state/caddy/certs`: local CA and server certificate used for HTTPS. Treat this as secret material.
- `deploy/backups`: timestamped backup sets

## Notes

- The frontend is served over HTTPS with a local CA and server certificate generated by `scripts/generate-tls-certs.sh`, so camera access can work once client devices trust `state/caddy/certs/smart-db-root-ca.crt`.
- The middleware should talk to Part-DB over the internal Docker network via `PARTDB_BASE_URL=http://partdb`, while `PARTDB_PUBLIC_BASE_URL` remains the external Caddy URL shown to users.
- Bearer tokens are redacted from middleware logs.
- The Part-DB container is pinned to the exact image digest that was verified on the target host, rather than a floating `latest` tag.
- The backup timer is scheduled explicitly in UTC, and the backup script takes a lock so timer-driven and manual runs do not overlap.
- Nightly backups intentionally exclude live `config/` and `state/caddy/certs/` unless `SMART_DB_BACKUP_INCLUDE_SECRETS=1` is set for an explicit disaster-recovery export.
